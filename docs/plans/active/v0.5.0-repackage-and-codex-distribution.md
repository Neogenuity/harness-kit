# v0.5.0 — Repackage to `plugins/harness-kit/` + Codex distribution

Status: **active**

## Objective

Move the distributed plugin from `plugin/` to `plugins/harness-kit/` and add
a Codex distribution channel, so the same source tree installs as a plugin
in both Claude Code and Codex.

## Value

Codex reads project skills from `.agents/skills/` today, and the kit
installs *into* Codex via the manual clone-and-copy path the README
documents — workable, but not versioned and not updatable in place. Codex
plugins (`.codex-plugin/plugin.json`, marketplaces at
`.agents/plugins/marketplace.json`, plugins conventionally under `plugins/`
— per Codex's plugin docs; re-verify and stamp at execution, see scope
item 8) are the versioned distribution channel, and their layout
conventions match a `plugins/harness-kit/` tree. One repackage buys a second
first-class install path with zero content duplication.

### Additional gaps from the PR #6 review (folded in, validated 2026-07-11)

A repo-wide Codex-plugin review against the current official docs surfaced
four items; each was reproduced/validated locally and folded into scope.

- **[P1] Git-root-resolve the generated Codex hook commands** (NEW scope —
  see the sequencing note in Decisions). `templates/providers/codex/hooks.json`
  runs all five hooks as `bash scripts/hooks/…`, i.e. relative to CWD. Codex
  invokes hooks with the *session* CWD, which may be a repo subdirectory, so a
  session started from `docs/` exits **127** (script not found) — reproduced
  locally; the Git-root-resolved form exits 0. Change all five commands to
  resolve from the Git root (e.g.
  `bash "$(git rev-parse --show-toplevel)/scripts/hooks/…"`, failing open when
  not in a repo) and add a nested-CWD regression test. This is a runtime bug
  in the *currently shipped* hooks, not repackage-specific. Ref:
  <https://learn.chatgpt.com/docs/hooks>.
- **[P1] Complete the Codex marketplace entry** (extends scope items 2 & 8).
  Beyond the local `source.path`, the current contract also wants a top-level
  `interface.displayName` and, per entry, `policy.installation`,
  `policy.authentication`, and `category`, with a `./`-prefixed `source.path`
  contained by the marketplace root. `interface.displayName` was not in the
  original item 2. Ref: <https://developers.openai.com/codex/plugins/build>.
- **[P1] Smoke-test install + activation, not just registration** (extends
  Verification item 4). `codex plugin marketplace add` only registers the
  catalog; it does not prove the plugin installs or exposes `$harness-kit`.
  Add: `codex plugin list --available --json` →
  `codex plugin add harness-kit@harness-kit` → `codex plugin list --json` → a
  new Codex session confirming the skill is visible and invocable. Also
  confirm the plugin appears **exactly once** — Codex reads both
  `.agents/plugins/marketplace.json` and the legacy-compatible
  `.claude-plugin/marketplace.json`, so a dedup check is required. Ref:
  <https://developers.openai.com/codex/plugins>.
- **[P2] Make the manifests gate semantic, not just JSON/version** (extends
  scope item 9). In addition to `jq empty` + VERSION equality, assert:
  plugin/marketplace **name** agreement; a semver-shaped version; `skills ==
  "./skills/"` resolving inside the plugin root; each marketplace `source.path`
  existing and contained by the marketplace root; required marketplace
  `policy`/`category` fields present; and both marketplace entries resolving to
  manifests carrying the expected version. Ref:
  <https://developers.openai.com/codex/plugins/build>.

## Decisions

- 2026-07-10 — Capture over waive (carry-over debt retired for real). Method:
  `codex exec --dangerously-bypass-hook-trust` in a scratch repo with a
  tee hook, driving one `cat` read + one apply_patch edit. Findings: (a) real
  apply_patch command is the bare `*** Begin Patch` … `*** End Patch` body,
  no `apply_patch` literal — the fixtures' wrapper form was wrong, which is
  why CI stayed green on a guard bypass; (b) `tool_name` carries the tool
  identity (`apply_patch` for edits, `Bash` for shell); (c) Stop payload has
  `stop_hook_active` + `last_assistant_message`; Stop output accepts
  `{"continue":true}` / `{}` — advise-once protocol validated. Resolution:
  fix in v0.4.1 (gate on the envelope marker), bare-envelope regression tests
  added. No `apply_patch|shell` alternation needed — the PreToolUse guards are
  matcher-less (see both), and `format.sh`'s `apply_patch` matcher matches the
  real `tool_name`.
