# Execution governance baseline (v0.10.0)

Status: active

## Objective

Ship the pull-forward-eligible baseline of the execution-governance plan:
an identity-carrying MCP trust inventory with a split-severity drift
check, the untrusted-content and risky-actions convention docs (the
risky-actions doc carries the documented safe-default posture), a
freshly-verified sandbox/approvals row in the provider matrix so those
docs only claim what a source proves, and supply-chain hardening of the
shipped CI template plus this repo's own workflows — while splitting the
high-tailoring-cost advanced profiles (per-provider sandbox/network
templates, audit-log export) into their own queued plan.

## Value

The kit's guards are deliberately advisory, and
[references/pattern.md](../../../plugins/harness-kit/skills/harness-kit/references/pattern.md)
already states that pre-tool interception is a guardrail, not a boundary.
Current practice treats execution containment as one combined control
system — sandbox + approvals + network policy — that should not be
scheduled last (<https://openai.com/index/running-codex-safely/>, validated
2026-07-10). The baseline below is documentation plus single-source drift
checks with near-zero per-project tailoring cost: it closes the
hostile-repo/prompt-injection guidance gap, gives MCP servers the same
single-source + drift-check treatment as `SECRET_PATTERNS` (extended to
server *identity*, because an allowed name repointed at different code is
the attack the inventory exists to catch), and stops the kit from
installing a CI workflow with a mutable third-party action and
default-write token permissions into other people's repos. The expensive
per-provider enforcement profiles keep their later slot.

This plan was adversarially design-reviewed by Codex gpt-5.6-sol before
implementation (verdict on the first draft: REDESIGN, 14 findings); all
14 were accepted in adapted form — see Progress and Decisions.

## Scope

1. **MCP trust inventory (identity-carrying)** — `harness.conf` gains a
   marked TAILOR block defining `MCP_ALLOWED_SERVERS`: a newline-delimited
   multi-line string; each non-empty, non-`#` line is
   `<name><whitespace><expected-identity-substring>`, where the identity
   substring is matched fixed-string (grep -F semantics, no globs, `set -f`
   discipline) against the server's configured command+args or URL. Both
   the template and this repo's installed copy gain the block (the
   installed copy keeps its `PROVIDER_MATRIX_DOC` tailoring; this repo has
   no MCP configs, so its inventory is set-but-empty — the strict-mode
   dogfood default). New check **#8c** in `check-harness.sh` — the
   #8/#8b single-source drift-check family — extracts, per the provider
   matrix's MCP row (verified 2026-07): `.mcp.json` and `.cursor/mcp.json`
   `mcpServers` entries (name = key; identity = `command` + joined `args`,
   or `url`), `opencode.json` `mcp` entries (same shape), and
   `.codex/config.toml` `[mcp_servers.*]` tables (best-effort header +
   `command`/`args`/`url` line scan; quoted table names unwrapped). An
   entry with an explicit `"disabled": true` / `"enabled": false` flag is
   skipped; an empty `mcpServers`/`mcp` map is silent (the shipped
   OpenCode template carries `"mcp": {}` — file existence must not warn).
   **Split severity:** (i) inventory variable unset + at least one parsed,
   enabled server → one WARN ("MCP servers configured but no trust
   inventory declared") — the adoption path; (ii) inventory set (even
   empty) + a parsed enabled server whose name is absent or whose identity
   does not contain its pinned substring → **ERROR** naming file, server,
   and which of name/identity failed — post-adoption drift is #8-class;
   (iii) an existing MCP config that cannot be parsed, or jq absent while
   JSON MCP configs exist → one WARN naming the unaudited file ("not
   audited" is never silent); (iv) no configs → silent regardless of
   inventory. Never blocks a turn; check-harness exit reflects ERRORs
   only, as today. Substring identity matching is drift *detection*, not
   equivalence proof — the check's comment says so, like #8's. *Acceptance:
   `test-check-harness.sh` gains an enumerated fixture block using the
   existing `assert_ok` / `assert_flags` / `assert_warns` /
   `assert_ok_without` helpers, asserting message needles (file + server
   name) and exit codes: (a) undeclared server → per-provider case for
   each of `.mcp.json`, `.cursor/mcp.json`, `opencode.json` (WARN when no
   inventory; ERROR when inventory set), and the TOML case incl. a quoted
   hyphenated table name; (b) name allowed but identity repointed → ERROR;
   (c) fully covered incl. names with dots/hyphens → clean; (d) empty
   `mcpServers` / `"mcp": {}` maps → silent; (e) disabled entry → silent;
   (f) no configs, no inventory → silent; (g) no-inventory WARN fires
   exactly once across multiple configs; (h) malformed JSON alongside a
   valid config → unaudited-WARN for the malformed file AND the valid
   file still checked; (i) jq removed via PATH shim with a JSON config
   present → unaudited WARN, exit 0; (j) same name in two providers with
   different identities, one pinned → the mismatching provider ERRORs.*
2. **Untrusted-content convention doc** — new template
   `templates/docs/conventions/untrusted-content.md`, hostile *inputs*
   only: tool output/repo/web content are data, not instructions; the
   untrusted-clone checklist (inspect before executing — direnv/`.envrc`,
   package-manager lifecycle scripts, git hooks; first pass read-only;
   run inside the provider sandbox with no secrets mounted); which layers

   prompts do not — implements pattern.md's

   servers as an input surface (pointer to the item-1 inventory).
   Cross-references risky-actions.md for the default posture. Template
   docs use backtick paths, not relative markdown links, for repo files
   (templates are not scanned by check #4; the installed dogfood copies
   are — see Decisions). *Acceptance: template exists; tailored dogfood
   copy at `docs/conventions/untrusted-content.md`; linked from this

   `AGENTS.md.tmpl`; `bash scripts/harness/check-harness` link gate green over
   the installed copies; `grep -L '](' templates/docs/conventions/*.md`
   confirms no relative md links in templates.*
3. **Risky-actions convention doc (+ default posture)** — new template
   `templates/docs/conventions/risky-actions.md`, risky *outputs*:
   destructive-command policy (history rewrites, `rm -rf`, bulk deletes,
   data-store drops), production-environment restrictions, and the
   **documented default posture** (workspace-only writes, no network by
   default, approvals on for destructive actions, with explicit loosening
   steps). Every wiring example is classified by layer: **pre-action
   enforcement** (provider-native permission denies, approval policies,
   sandbox settings — citing the item-4 matrix row), **in-turn advisory
   feedback** (guard-config.sh file-edit denial — file edits only, shell

## Out of scope

The advanced profiles themselves (sandbox/network templates,
devcontainer, audit-log export — now `execution-sandbox-profiles.md`,
queued); any enforcement mechanism of the kit's own
(configure-don't-rebuild); identity *equivalence* proof for MCP servers
(the check detects drift by fixed-string identity substring, exactly as
honest as #8's substring discipline — full normalization is the
sandbox-profiles plan's territory if ever); full TOML parsing (documented
best-effort scan); CI secrets management and org-level identity
(provider-managed); recording eval baselines
([eval-discrimination.md](v0.12.0-eval-discrimination.md)'s territory).

## Decisions

- 2026-07-11 — **Split severity for #8c** (revised from always-WARN
  after sol review): no-inventory adoption is WARN (configs predate the
  inventory by construction); once `MCP_ALLOWED_SERVERS` is declared,
  an uncovered or identity-mismatched enabled server is ERROR — the
  same "drift from a declared single source is a silent hole" logic as
  #8/#8b; an unauditable config is a WARN, never silence (a governance
  check whose failure mode is indistinguishable from a clean pass
  audits nothing).
- 2026-07-11 — **Identity substrings over bare names** (sol blocker):
  name-only allowlisting cannot see a repointed server. Fixed-string
  substring matching against command+args/URL keeps the check portable
  and honest ("detects drift, not equivalence" — check #8's own
  discipline) without a normalization engine.
- 2026-07-11 — **Posture lives in risky-actions.md** (revised): the
  plan's own split is inputs (untrusted-content) vs. outputs
  (risky-actions); workspace writes, network egress, and approvals
  govern outputs. untrusted-content.md cross-references it.
- 2026-07-11 — **Matrix row now, not later**: enforcement claims in
  shipped docs must cite verified facts (ADR 004); a posture doc
  pointing at an unverified enforcement layer would be exactly the
  "claims a layer it doesn't have" failure the acceptance forbids.
- 2026-07-11 — **CI workflows join PROTECTED_PATHS** (template
  default): CI is the enforcing layer for shell-level edits the guards
  deliberately don't scan; leaving the workflow that runs the gates
  agent-editable was an inconsistency. Escape hatch unchanged
  (`HARNESS_ALLOW_MECHANISM_EDITS=1`).
- 2026-07-11 — **Check numbering: #8c drift family, #10d doctor**: the
  inventory check is single-source drift (#8 family) with the split
  severity above; the mutable-`uses:`-ref scan is a freshness/hygiene
  WARN (doctor family; #10d because #10c was already the matrix-freshness
  check).
- 2026-07-11 — **Pin to latest v4 of actions/checkout, not v5**: the
  pin is a supply-chain fix, not an upgrade; changing majors would
  couple the hardening to behavior changes this plan doesn't need.
