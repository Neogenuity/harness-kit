# Outcome telemetry + doc gardening

Status: **completed** — v0.17.0 (released 2026-07-15)

## Objective

Grow the local harness log from friction-only events into a backward-compatible
outcome stream, make `verify.sh` gate results mechanically trendable, and add a
doc-gardening workflow that finds knowledge-base drift without silently
modifying a repository. An audit should be able to say which gates fail or
repeat, which paths repeatedly trigger guardrails, which plan/review sessions
can be correlated locally, and which documentation needs attention—while
remaining honest when the required local history or metadata is unavailable.

## Value

`.harness/log.jsonl` currently records denials, advisories, lint findings, and
v1-shaped review findings, but it cannot answer whether the executable
definition of done passed or where verification repeatedly fails. The existing
74-line dogfood log is useful evidence that mixed event families already exist;
it is not a CI fixture or a project-wide telemetry backend.

This release completes the last queued mechanism theme before 1.0. Gates,
reviews, evals, plans, and provider-observability boundaries now exist, so the
kit has stable outcome sources worth aggregating. Doc gardening closes the
parallel knowledge-base loop: existing doctor checks cover links in `AGENTS.md`
and `docs/**/*.md` plus freshness for one configured matrix, but root Markdown,
non-link path references, repo-wide verification stamps, and scheduled human
review remain unowned.

## Scope

1. **Freeze a mixed-v1/v2 local event contract.** Define an explicit
   `version: 2` envelope and event-specific fields before any producer or
   consumer changes. Record field type, provenance, null/absence semantics,
   and privacy classification. Session, provider, skill, plan, token/cost, and
   retry data are optional and accepted only from an explicit producer or a
   documented deterministic derivation—never guessed from provider files or
   imported automatically from provider telemetry. Multiple active plans are
   represented without choosing one arbitrarily. Existing reviewer findings
   remain exact five-key v1 lines; old logs are never rewritten.
   *Acceptance: a checked-in fixture and parser tests accept valid v1, valid v2,
   interleaved v1/v2, absent optional identifiers, and unknown/malformed rows;
   the existing seeded-defect review schema stays green; raw prompts, commands,
   tool output, credentials, endpoints, and auth headers are absent from every
   v2 fixture and producer.*
2. **Emit fail-open verification outcomes.** Instrument shipped `verify.sh` so
   every gate actually run emits exactly one local event with stable gate label,
   fast/full mode, pass/fail outcome, and nonnegative portable duration. Serial
   early failures must log before exit; parallel events must remain complete and
   race-safe; skipped full gates under `--fast` are not fabricated as runs.
   Event writing never captures command arguments/output and never changes gate
   stdout, ordering, cleanup, or exit status.
   *Acceptance: deterministic tests cover serial pass/fail, early abort,
   parallel pass/fail, final-barrier reaping, `--fast`, concurrent appends,
   missing `jq`, disabled logging, malformed/unwritable log targets, and prove
   byte-stable user-facing output plus unchanged gate exit behavior when
   telemetry fails.*
3. **Ship deterministic trend reduction and audit guidance.** Add a portable,
   tested reducer for mixed logs and make audit mode consume its results rather
   than perform free-form arithmetic. Define UTC windows, stable sort order,
   invalid-row accounting, zero/single-sample output, gate-failure denominators,
   repeat-deny keys, review-finding counts, and retry derivation. Do not derive
   plan-cycle time from Git directory transitions: free-form Progress prose and
   rename history do not provide a portable, reliable lifecycle timestamp, so
   v0.17.0 reports it N/A. Join sessions to commits locally through the
   `Harness-Session-Id:` trailer. PR enrichment stays N/A until an explicit,
   versioned metadata producer exists. Eval drift comes from the existing
   baseline and result artifacts, not from provider telemetry. The durable
   correlation foundation is exact session-trailer→commit attribution rather
   than an invented orphan map format.
   *Acceptance: a deterministic golden fixture pins the rendered table and
   recommendation order for mixed versions, gate failures/retries, repeat-deny
   paths, review findings, explicit plan-cycle N/A output, trailer joins, eval
   drift, bad rows, no data, no Git, and shallow Git; a manual dogfood run
   summarizes the real ignored log but CI never depends on it or on network
   access.*
4. **Add the canonical `doc-garden` skill.** Ship a self-contained skill that
   reuses existing harness checks and adds root/non-`docs` Markdown, local
   anchors, prose/code path references, and repo-wide verified-date review.
   External-link probing is a separately authorized, capability-checked skill
   step rather than portable mechanism. Ignore fenced examples and explicitly
   future/queued paths according to a documented rule.
   The default action is a report or proposed patch; commit, push, and PR
   creation require explicit user authorization and available provider tooling.
   Include provider-neutral scheduled-run guidance, not a daemon or an
   unstamped provider-specific cron claim.
   *Acceptance: a seeded fixture catches a root dead link, missing anchor,
   deleted-path reference, stale and malformed stamps, while accepting fresh
   stamps, fenced examples, future planned paths, and an offline external URL;
   negative cases prove the skill neither mutates nor publishes without
   authorization. Canonical docs, AGENTS links, and generated provider stubs
   pass `sync-agent-skills.sh --check`.*
5. **Define adoption and migration behavior.** Update init/audit/router guidance
   to describe the v2 local stream and doc-garden offer. Existing installs gain
   new mechanism files through normal update inventory and keep v1 data
   readable; content and the new skill are proposed, never auto-authored or
   overwritten during update. Preserve the v0.16 provider-observability privacy
   floor: no collector, provider export, automatic cross-stream correlation,
   repo-stored secret, or raw-content capture.
   *Acceptance: fresh-install, v0.16-style mixed-log update, no-op second update,
   declined-skill, and tailored-file fixtures pass; the migration note states
   exactly which files update mode replaces, adds, diffs, or leaves untouched.*
6. **Behavioral and release evidence.** Add a seeded doc-garden behavioral task
   only if deterministic fixtures cannot prove instruction-following and
   authorization boundaries. Any paid Terra/Luna provider run requires separate
   maintainer approval and is evidence, not a prerequisite for deterministic
   implementation. Complete template-first dogfood rollout, reciprocal review,
   post-gate inferential review, and the release workflow.
   *Acceptance: every focused suite and `bash scripts/verify.sh` pass after
   integration; all confirmed review findings are fixed and re-run; VERSION,
   both plugin manifests, CHANGELOG, and the dogfood manifest follow the release
   skill; the release commit passes a final full gate before annotated
   `v0.17.0` tagging.*

## Out of scope

Hosted dashboards, collectors, storage/retention services, OTel configuration,
automatic provider-session ingestion, raw prompt/tool capture, organization
identity or cost accounting, and autonomous harness self-modification. v0.17.0
does not promise global or team-wide trends from a git-ignored local file, infer
a provider/skill/plan when the producer did not supply one, scrape review prose
for correlation, require GitHub or network access for deterministic checks, or
perform the remaining launch-readiness maintainer actions.

## Delegation and integration

The lead freezes interfaces and owns this plan/roadmap, root dogfood files,
generated artifacts, version trio, changelog, manifest, integration, release,
and final adjudication. Terra and Luna use disjoint single-writer paths; neither
edits installed root copies or generated stubs.

1. **Joint interface review, lead decision.** Luna drafts the v2 field-
   provenance/privacy contract and exact audit output semantics. Terra challenges
   it against Bash 3.2/BSD portability, concurrent JSONL writes, current hook
   payloads, and `verify.sh` failure paths. The lead records the frozen event,
   attribution, retry, reducer, and authorization contracts before either
   implementation stream begins.
2. **Terra — mechanism and deterministic evidence (single writer).** Own the
   shipped v2 writer/parser/reducer, hook compatibility, `verify.sh`
   instrumentation, any narrowly justified doc-scanner mechanism,
   install/update mechanism inventory, and focused shell fixtures under
   `plugins/harness-kit/skills/harness-kit/templates/scripts/**`. Implement only
   frozen fields; return semantic ambiguity to the lead rather than inferring
   provider, skill, plan, or PR identity.
3. **Luna — content, workflow, and behavioral evidence (single writer).** Own
   schema/audit explanations, router and init/audit/update guidance, the
   canonical/template `doc-garden` skill, AGENTS template links,
   provider-neutral scheduling and migration guidance, and any seeded
   behavioral task/grader/reference. Do not duplicate current link/stamp checks
   or claim provider features without a fresh primary-source stamp.
4. **Reciprocal read-only review.** Terra reviews Luna for installability,
   update non-clobbering, duplication of existing doctor checks, offline
   determinism, and executable acceptance criteria. Luna reviews Terra for v1
   compatibility, schema meaning, privacy leakage, trend math, missing/N/A
   behavior, and preservation of gate semantics. Each author fixes confirmed
   findings only in owned files.
5. **Lead integration review.** The lead independently reproduces every finding,
   integrates both streams, rolls templates into the root dogfood harness,
   syncs generated stubs, re-pins the manifest, and runs focused and full gates.
   Only after deterministic green does the lead delegate the integrated diff to
   the canonical `code-reviewer`, fix confirmed findings, and repeat the
   relevant gates. The lead alone prepares and tags the release.

## Decisions

- 2026-07-15 — **v0.17.0 is the correct activation version:** shipped hook,
  verification, reducer, update-inventory, and skill/template behavior changes
  require at least a minor release before 1.0.
- 2026-07-15 — **Mixed versions are the compatibility contract:** v2 producers
  add a versioned envelope; v1 hook and reviewer records remain readable and
  review findings keep their exact five-key shape.
- 2026-07-15 — **Unknown attribution stays unknown:** optional identifiers come
  from an explicit producer or a documented deterministic derivation; multiple
  active plans, absent metadata, shallow Git, and unavailable PR context report
  null/N/A rather than a guess.
- 2026-07-15 — **Local evidence is not global telemetry:** the git-ignored log
  and available Git/eval artifacts support local trends only. Provider telemetry
  stays separate and privacy-preserving.
- 2026-07-15 — **Deterministic reducer before prose trends:** audit consumes a
  tested result so rates, retries, sparse data, and malformed rows do not depend
  on model arithmetic.
- 2026-07-15 — **Doc garden extends rather than duplicates the doctor:** it owns
  uncovered root/non-doc/path/stamp surfaces and authorization-aware remediation.
- 2026-07-15 — **Delegation keeps single writers:** Terra owns mechanism and
  deterministic assurance; Luna owns content/workflow and behavioral evidence;
  reviews swap areas; the lead alone integrates root dogfood and releases.
- 2026-07-15 — **Deterministic fixtures are sufficient for doc-garden:** link,
  anchor, history, stamp, fence, inline-comment, planned-marker, no-mutation, and
  failure cases directly pin the mechanism and authorization boundary, so an
  additional behavioral task would duplicate evidence and no paid run is needed.
- 2026-07-15 — **The frozen v2 envelope is
  `{version,ts,hook,event,file,detail,context,data}`:** all eight keys are
  present; `context` and `data` carry typed extensions; reviewer findings remain
  exact v1. Gate data includes name, mode, outcome, exit code, and integer
  duration. Optional attribution is explicit-only and records its source.
- 2026-07-15 — **Correlate only what has a producer:** session trailers join to
  commits; retry episodes require a session id; plan cycles and PR mapping stay
  N/A/deferred until a versioned lifecycle/PR metadata source exists. External
  link probing is an authorized skill action, not a Bash/jq/git mechanism
  dependency.
- 2026-07-10 — **Human-in-the-loop stays:** audit recommends and people decide;
  autonomous harness evolution is explicitly out of scope for 1.0.
