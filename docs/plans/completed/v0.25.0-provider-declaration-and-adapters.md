# v0.25.0 — Single provider declaration + generated adapters

Status: **completed** (v0.25.0, 2026-07-23) — Phase 5 of
[../standard-consumer-layout.md](../standard-consumer-layout.md)

## Objective

Collapse the four hand-curated provider lists in `harness.conf`
(`PROVIDERS`, `HOOK_WIRED_PROVIDERS`, `AGENT_PROVIDERS`,
`EXECUTION_PROFILE_PROVIDERS`) to **one** adopter declaration —
`HARNESS_PROVIDERS` — by moving the per-provider capability facts that decide
each list's membership into a kit-owned table
(`scripts/harness/lib/provider-caps`). Grow `sync` so it emits the two
derived artifacts the standard layout reserves space for: per-provider
`.harness/adapters/<provider>.md` wiring summaries, and the native
secret-deny mirrors (`SECRET_PATTERNS` → `.claude/settings.json` deny /
`opencode.json` read-denies) with a semantic `--check`.

## Value

Correct membership of the four lists is **kit knowledge, not repo policy**:
whether `.codex` reads `.agents/` natively (so it needs no skill stub),
whether `.opencode` has a bash hook shim (so it can be hook-wired), which
dialect each provider's agent stub takes. Today an adopter has to know all
of it to fill four lists correctly, and the same facts are re-encoded in
`sync-lib.sh`, `check-instructions.sh`, and three `harness.conf` comments.
One declaration + one kit-owned table means the adopter says *which
providers I use* once, the kit derives the rest, and the facts live in
exactly one place — the last of the "ownership encoded five ways" cleanups
before the layout freezes for launch. Done now it is a one-repo re-pin; done
after launch it is a breaking `harness.conf` migration for every adopter.

## Design

### Capability table (mechanism)

`scripts/harness/lib/provider-caps` — plain text, bash-3.2 `read`-parseable,
no jq (ADR 002), sha-pinned + guard-protected like `kit-manifest`. One row
per provider dir; whitespace-separated columns:

```
# <provider> <skill_stubs> <agent_dialect> <hook_config> <exec_config>
.claude    yes  md           .claude/settings.json:nested  .claude/settings.json:claude
.cursor    yes  md           .cursor/hooks.json:flat       .cursor/sandbox.json:cursor
.codex     no   toml         .codex/hooks.json:nested      .codex/config.toml:codex
.opencode  yes  md-subagent  none                          opencode.json:opencode
```

- `skill_stubs` — `yes|no`. `no` ⇔ the provider reads `.agents/skills/`
  natively (`.codex`), so it is excluded from generated skill stubs.
- `agent_dialect` — `md | toml | md-subagent | none`: how the generated
  agent stub renders (`.codex` = TOML, `.opencode` = Markdown + `mode:
  subagent`), or `none` for a provider that receives no agent stub.
- `hook_config` — `<path>:<shape>|none`, `shape ∈ {nested,flat}`; `none`
  means no bash hook shim (`.opencode` is descoped, per the frozen matrix).
- `exec_config` — `<path>:<checker>|none`; `checker` selects the #8e
  validator (`claude|cursor|codex|opencode`).

`.agents` is **not** a provider row — it is the canonical home, never a
wiring target. The table is the source for the tuple/exec knowledge that
`check-instructions.sh` currently hard-codes; that check keeps its frozen
tuple tables (the hook event/matcher contract is richer than a table cell)
but reads the config path/shape/checker selector from the caps table so the
two never disagree about *which file* a provider wires.

### Derivation + override (`provider-lib.sh`, mechanism)

`scripts/harness/lib/provider-lib.sh` reads the table and exposes
`harness_caps_field`, `harness_caps_providers` (every known provider — the
validation universe), and `harness_derive_providers <facet>`
(`skill|agent|hook`) = `HARNESS_PROVIDERS` filtered by that facet's
capability. Sourced by `check-common.sh` (so every check family sees the
derived sets) and by `sync-lib.sh`.

Resolution per list, preserving the **declared-not-inferred** invariant
(ADR 008/010) and the existing set-vs-unset migration diagnostics:

1. explicit `harness.conf` value set → use it verbatim (the override knob);
2. else `HARNESS_PROVIDERS` set → derive from the table;
3. else leave **unset** → the current loud "declare the set" diagnostics
   fire (a legacy pre-declaration install is diff-only, so neither variable
   appears on its own).

The derived sets reproduce today's dogfood lists exactly
(`HARNESS_PROVIDERS=".claude .cursor .codex .opencode"` →
skills `.claude .cursor .opencode`, agents all four, hooks
`.claude .cursor .codex`), so this repo's behavior is unchanged.

### Execution profiles stay explicit opt-in (the deliberate asymmetry)

`EXECUTION_PROFILE_PROVIDERS` does **not** derive from `HARNESS_PROVIDERS`.
Skill/agent/hook wiring is non-invasive registration; execution profiles
impose strict runtime sandbox floors (default-deny network, empty writable
roots) that routinely conflict with local dev — auto-adopting them as a side
effect of naming your providers would be a footgun and reverses the
deliberate opt-in of ADR 008. So the knob stays explicit and unset-by-default
(unadopted = clean pass). What Phase 5 *does* change: its per-provider config
path and validator move to the caps table like the others, and the declared
set is validated to be a subset of `HARNESS_PROVIDERS ∩ {exec-capable}` — you
cannot adopt an execution floor for a provider you do not wire. This
asymmetry is recorded and justified in ADR 011.

### `sync` grows two derived artifacts

- **Adapters.** `bash scripts/harness/sync` writes
  `.harness/adapters/<provider>.md` for each wired provider — a generated
  wiring summary (skill-stub home, agent dialect, hook config, exec config,
  native-`.agents/` note) rendered from the caps table + the derived sets.
  `sync --check` pins them exactly like skill/agent stubs (committed,
  drift-checked, no `kit-manifest` entry — generated *into* the repo, not
  shipped *from* the kit). Orphan adapters (a provider dropped from
  `HARNESS_PROVIDERS`) fail `--check` in both modes.
- **Secret-deny mirrors.** New subcommand `bash scripts/harness/sync secrets
  [--check]`, **jq hard-required for this subcommand only** (it writes JSON).
  Generates the native deny lists from `SECRET_PATTERNS`: `.claude/settings.json`
  `permissions.deny` gets `Read(<pat>)` entries (**deny-only** — the Claude
  platform resolves deny-beats-allow, so no allow list is emitted; the
  asymmetry with OpenCode is documented), `opencode.json` `permission.read`
  gets `<pat>: "deny"`. jq **set-reconciliation**: only the secret-derived
  entries are added/removed; every other key and every non-secret deny the
  repo owns is preserved. `--check` is **semantic** (compares the *set* of
  secret entries, not byte layout) and exits non-zero on drift. Checks
  `#8`/`#8b` remain as independent substring verification and now name
  `sync secrets` as the fix.

## Scope

- [x] **Slice 1 — caps table + derivation + single declaration.**
  `provider-caps` (mechanism data) + `provider-lib.sh` (derivation fns);
  `check-common.sh` and `sync-lib.sh` derive the three wiring lists with
  explicit-override + declared-not-inferred semantics; `check-instructions.sh`
  reads config path/shape/checker from the table (tuple tables stay);
  `EXECUTION_PROFILE_PROVIDERS` subset-validated against the table; install
  migration proposes `HARNESS_PROVIDERS` (union of declared sets) once;
  `kit-manifest` pins the two new mechanism files; dogfood `harness.conf`
  switches to the single declaration. *Acceptance: derived sets equal the
  prior explicit sets; checks `#8d`/`#8e` gate on the derived/validated sets;
  full verify green.*
- [x] **Slice 2 — generated adapters.** `sync` write + `--check` for
  `.harness/adapters/<provider>.md`; check families follow. *Acceptance:
  `sync --check` clean; a dropped provider's orphan adapter fails; doc-garden
  clean on the new dir.*
- [x] **Slice 3 — secret-deny mirror generation.** `sync secrets [--check]`
  (jq-hard); reconciliation preserves other keys; semantic `--check`; `#8`/`#8b`
  point at it. *Acceptance: generating then `--check` is clean; a hand-removed
  deny entry is regenerated; a non-secret deny survives reconciliation.*
- [x] **Slice 4 — ADR 011 + docs + release.** ADR 011 (capability table,
  derived wiring, the exec opt-in exception, generated adapters, secret
  mirrors, the Claude deny-only asymmetry); update `provider-matrix.md`
  (caps-table source-of-truth stamp), `references/pattern.md`, `AGENTS.md`,
  `SKILL.md` + `references/modes/{init,update}.md`; update-from-v0.24.0
  fixture; release v0.25.0. *Acceptance: link checker + stub checks + eval
  machinery green against the new tree; this repo migrated identically.*

## Out of scope

- Inferring any provider set from disk — declarations stay explicit; only the
  *derived* facts (capability) move to kit data. `HARNESS_PROVIDERS` and
  `EXECUTION_PROFILE_PROVIDERS` are the two things the adopter still states.
- Making `provider-caps` YAML/TOML/JSON (ADR 002: bash-3.2 + no-jq parse; jq
  only in `sync secrets`, which writes JSON).
- A third native-config generator beyond the two secret-deny mirrors (e.g.
  generating the execution-profile floors) — Phase 5 generates the
  secret-deny lists only; the execution floors stay hand-authored + `#8e`-checked.
- Auto-deleting the legacy four lists from an adopter's tailored `harness.conf`
  on update (diff-only policy) — update proposes `HARNESS_PROVIDERS`; the
  redundant lists remain as harmless overrides, optionally WARNed as removable.
