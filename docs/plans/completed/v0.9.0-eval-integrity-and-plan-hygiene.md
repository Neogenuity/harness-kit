# Eval integrity and plan hygiene

Status: shipped (v0.9.0)

## Objective

Close the integrity and hygiene holes a 2026-07-11 project review found in
the v0.8.0 eval layer and the plans/docs surface, so the recorded eval
baselines are trustworthy and the roadmap owns all known follow-up work.

## Value

The eval layer is the repo's measurement instrument — every later claim that
a harness change improved (or regressed) agent behavior is only as credible
as the scorer underneath it. The review found four *verified* integrity
gaps that let it silently mis-score: lexicographic (not chronological)
latest-run selection, baselines built from uneven trial counts, no
enforcement that `--provider mock` rows stay out of recorded baselines, and
`eval.sh` measuring a dirty working tree without saying so (each trial
clones committed `HEAD`, so uncommitted changes are silently invisible to
the agent). None of these were caught by `test-eval.sh` because they are
properties of the scorer's *aggregation* and *invocation* discipline, not of
any single task's grader.

The findings and their proposed remediation were adversarially reviewed by a
second model (Codex gpt-5.6-sol) before implementation started; three of the
originally-proposed fixes were redesigned as a result (see Decisions and
Progress). Implementation is split across two parallel agents — one on the
eval mechanism itself, one on this plan and the surrounding docs/CI — so
this plan is also the coordination record for that split.

## Scope

1. **Latest-run selection by timestamp** — `eval-harness.sh`'s "latest run"
   selection moves from a lexicographic sort of the run-id string to
   `max_by([(.run_started_at // 0), .run])` over each run's `results.jsonl`,
   with a legacy fallback (missing `run_started_at` sorts as epoch 0) so old
   result lines keep parsing. Previously a custom `--run-id` that happened to
   sort later as a string could permanently outrank a chronologically newer
   timestamped run, regardless of when it actually executed.
   *Acceptance: a seeded scorer fixture in `test-eval.sh` proves a
   lexicographically-earlier but chronologically-later run is selected over a
   lexicographically-later but chronologically-earlier one.*
2. **Results-dir collision refusal + collision-resistant default run ids** —
   `eval.sh` refuses to reuse a `--run-id` whose results directory already
   has recorded results, and the default (unsupplied) run id is suffixed with
   provider/model so two different provider runs sharing a timestamp can't
   collide. *Acceptance: reusing a run id that already has recorded results
   dies with a clear error instead of silently merging or overwriting —
   proven by `test-eval.sh`'s runner-guards fixture block (sub-case (d),
   results-dir collision refusal), which runs `eval.sh` itself as a
   subprocess against a throwaway fixture repo rather than asserting on it
   indirectly.*
3. **Dirty-tree refusal** — each trial's workspace is a throwaway clone of
   committed `HEAD`, so running `eval.sh` against a dirty working tree
   silently measures whatever was last committed, not the change actually
   staged or unstaged on disk. `eval.sh` now refuses to run against a dirty
   tree unless `--allow-dirty-head` is passed explicitly, converting a silent
   measurement error into a required acknowledgment.
   *Acceptance: `eval.sh` dies on a dirty tree without the flag;
   `test-eval.sh`'s runner-guards fixture block (sub-case (a), dirty-tree
   refusal) proves both the refusal and the `--allow-dirty-head` override by
   running `eval.sh` itself against a throwaway fixture repo.*
4. **Negative-task outcomes: `negative_violation` vs `task_failure`** —
   `check.sh` gains an exit-3 convention for negative tasks: exit 3 means
   "the forbidden shortcut was taken," recorded as a distinct
   `negative_violation` outcome; a plain exit 1 keeps its existing meaning,
   "the agent didn't reach the real goal" (`task_failure`). `eval-harness.sh`
   fails loudly on any `negative_violation` outcome regardless of which suite
   the task belongs to. *Acceptance: `test-eval.sh` proves the existing
   negative task's `reference/violate*.sh` fixtures score `negative_violation`;
   a seeded `eval-harness.sh` fixture proves a `negative_violation` row exits
   1; and the runner-guards fixture block's end-to-end case (sub-case (e))
   proves the full pipeline — `eval.sh` running a negative task's mock
   reference solution through `check.sh`'s exit-3 convention — records
   `outcome == "negative_violation"`, `pass == false`, and exits 0 itself (a
   violation is a recorded outcome, not a runner error), with
   `eval-harness.sh` then failing loudly on it.*
5. **Baseline integrity** — `--update-baseline` excludes `--provider mock`
   rows from recorded pass@k/pass^k (mock rows exercise the reference
   solution, not agent behavior, and were previously eligible to be baked
   into a baseline), refuses atomically unless every scored cell has exactly
   `--expected-trials` trials (default 3, so a partial or over-long run can't
   quietly become the new baseline), and writes each cell's `recorded` date
   from the run's own timestamp rather than wall-clock time at update.
   *Acceptance: seeded fixtures in `test-eval.sh` cover mock exclusion, the
   atomic refusal on an uneven trial count, and the derived-date behavior.*
6. **Task-metadata enum validation + `provider:` gate** — `TASK.md`'s
   `suite` / `polarity` / `provider` / `grade` fields are validated against
   their enums at load time (a typo now dies loudly instead of silently
   falling back to a default), and a `provider:` value gates a task off a
   provider it doesn't target; the `mock` provider is exempt from the gate
   since it always exercises the reference solution regardless of the
   declared provider. *Acceptance: an invalid metadata value dies at load
   time; the existing task bank validates clean offline via `test-eval.sh`.*
7. **Plans hygiene** — [docs/plans/active/.gitkeep](../active/.gitkeep) is tracked (a
   fresh clone previously lost the `active/` directory entirely, so
   `scripts/harness/hooks/session-context.sh` had nothing to announce, and the kit's
   own SKILL init step requires the directory to exist); two orphaned
   follow-ups from v0.8.0's Next-action note are captured as queued plans —
   [eval-discrimination.md](v0.12.0-eval-discrimination.md) (a task bank that
   actually discriminates model behavior) and
   [launch-readiness.md](../active/launch-readiness.md) (the launch, currently
   tracked only as README checkboxes) — and the roadmap is re-sorted to
   include both. *Acceptance: `git ls-files docs/plans/active/` is
   non-empty; every link in [../README.md](../README.md)'s roadmap table
   resolves.*
8. **CI runs `verify.sh` directly** — `.github/workflows/ci.yml`'s separate
   "Regression tests" and ShellCheck-run steps collapse into a single step
   invoking `bash scripts/harness/verify` (after the ShellCheck-install step,
   since `verify.sh`'s first gate needs the binary on `PATH`), so
   [../../architecture/ARCHITECTURE.md](../../architecture/ARCHITECTURE.md)'s
   claim that "CI runs the same gates" becomes structurally true instead of
   hand-reconstructed step-by-step — a gate added to `verify.sh` can no
   longer silently fail to appear in CI. *Acceptance: `ci.yml` contains a
   `bash scripts/harness/verify` step, and the gate list in `verify.sh` is the
   only place CI's pass/fail criteria are defined.*
9. **README / release-skill version-staleness fix** — README's Status
   section drops the hardcoded `v0.6.0` claim (it had already survived two
   releases unnoticed) in favor of pointing at
   `plugins/harness-kit/VERSION` and the changelog, and
   [../../skills/release/SKILL.md](../../skills/release/SKILL.md) gains a
   sweep step and a verification bullet so a future release owns catching
   this instead of it silently going stale again. *Acceptance: README
   carries no hardcoded kit version; the release skill's Steps and
   Verification sections both name the check.*

## Decisions

- 2026-07-11 — Negative-task violations are a distinct outcome
  (`negative_violation`), not a suite change, because a failed negative task
  does not by itself imply reward hacking — the grader also fails on honest
  incompletion (`task_failure`), and collapsing both into "the negative suite
  failed" would have hidden which one actually happened.
- 2026-07-11 — The regression rule stays the absolute pass^k=1 invariant
  (any drop below it exits non-zero); baseline-vs-recorded comparison stays
  informational context on top of that invariant, not a replacement for it.
