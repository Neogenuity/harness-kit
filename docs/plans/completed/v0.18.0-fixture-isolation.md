# v0.18.0 — Fixture isolation

Status: **shipped 2026-07-16** (tag v0.18.0) — scoped, built, and verified in one
session from a launch-readiness review that reproduced the defect twice,
in-session, on this repo's own `main`.

## Objective

Close a defect class in the shipped regression tests: when `mktemp` fails, an
unguarded fixture does not abort — it runs its `git init`/`add`/`commit` in the
**host repository**. Guard every scratch allocation, harden every consumption
site, add the CI gate that makes the pattern unreintroducible, and pin the
behavior with a test that fails on the pre-fix tree. The kit ships this code
into other people's repos; a harness that damages the repo it protects fails at
its one job.

## Value

This is the only defect found to date where **harness-kit itself violates its
own core promise**. It is not theoretical and it is not rare in the environment
that matters:

- It fired **twice in one session** (2026-07-16), putting four junk commits on
  this repo's `main` (`ffc25cf`, `0d8fd06`, `fd1a112`, `0336748`; recovered by
  `git reset --hard 792d46a`, reflog-recoverable, none pushed).
- **CI cannot see it.** The trigger is `mktemp` *failing*, and on a stock runner
  it does not: Linux resolves `$TMPDIR`/`/tmp` (writable), and a stock macOS
  runner has a writable `/var/folders`. The class stays latent wherever the temp
  dir works, and detonates only where it is denied — a sandbox, a hardened
  runner. That is exactly where coding agents run, so the kit's own audience is
  the population at risk while its CI stays green.
- **shellcheck structurally cannot catch it** (verified: shellcheck exits 0,
  clean, on the archetype). Every variable involved is *correctly quoted* —
  quoting is precisely what makes `cd ""` a well-formed no-op instead of an
  error. The existing lint gate was never going to find this.
- A shipped reference doc, [fixture-recipe.md](../../../plugins/harness-kit/skills/harness-kit/references/fixture-recipe.md),
  **taught the anti-pattern as a recipe** — propagating the defect into human
  hands, in a snippet whose `git add -A && git commit` runs in whatever repo the
  reader is standing in.

## The mechanism (each step verified in-session, not assumed)

1. Bare `mktemp -d` on macOS resolves `_CS_DARWIN_USER_TEMP_DIR` (`/var/folders/…`)
   and **ignores `$TMPDIR`** — so it fails outright wherever only `$TMPDIR` is
   writable.
2. `WORK=$(mktemp -d)` with no guard swallows that rc=1 and leaves `WORK` **empty**.
   These suites run `set -uo pipefail` — no `-e`. `set -u` does **not** fire:
   the variable is assigned-but-empty, not unset.
3. `cd ""` is a **silent rc=0 no-op** that stays in the current directory. So
   `cd "$WORK" || exit 1` — a guard that *looks* sufficient — passes.
4. `git -C ""` is the same hazard wearing a seatbelt: verified, it resolves to
   the real repo and returns 0.

Net: `( cd "$WORK" && git init -q && git add -A && git commit )` commits the
developer's working tree onto their checked-out branch.

## Scope

1. **Guard every scratch allocation** — 26 sites in `scripts/` and 26 in the
   template twins, rewritten to the idiom the repo already established at
   `scripts/eval.sh:337` (whose comment names this exact hazard) and
   `scripts/test-audit-log.sh:4`. Top-level tests take one guarded base;
   fixtures carve subdirectories out of it, so a per-fixture `mktemp` cannot
   fail loose. *Acceptance: no bare `mktemp -d` remains in either tree; check
   #5b reports zero findings.*
2. **Harden every consumption site** — `${VAR:?}` at each `cd`/`git`/`rm` that
   consumes a fixture root. This is the second line of defense and it is not
   redundant: `|| return 1` inside a `w=$(make_fixture)` command substitution
   returns from the **subshell**, so the caller still receives `""` and its own
   unguarded `cd "$F"` was still live (`test-install.sh:162`, a `git commit`).
   *Acceptance: no unguarded `cd "$VAR"` into a fixture root remains.*
3. **check #5b — the CI gate** — fails the build on a `mktemp` in the scripts
   check #6 runs that lacks *both* an explicit `XXXXXX` template and a failure
   guard. Placed before #6 deliberately: a static gate on a file set must run
   before the gate that executes it. Scope is exactly what #6 runs — an
   adopter's own `scripts/deploy.sh` is theirs. Declared exceptions use a
   trailing `# harness-mktemp-ok`, matching the manifest's `# tailored` stance.
   *Acceptance: fires on a reintroduced bare `mktemp -d`; zero findings on the
   fixed corpus; no false positive on the four canonical safe forms.*
4. **`test-fixture-isolation.sh` — the behavioral proof** — runs every sibling
   suite from inside a throwaway canary repo under a `mktemp` shimmed to fail,
   and asserts the canary's HEAD and porcelain are untouched. Mechanism-agnostic
   (it does not care *how* a suite leaks, because `cd ""` lands the leak in the
   CWD). Two modes: `all` pins the base guard; `nested` grants the first
   allocation and fails the rest, which is what exercises the `${VAR:?}`
   carve-out defense. *Acceptance: passes on the fixed tree; **fails on the
   pre-fix tree**.*
5. **Stop teaching the bug** — rewrite `fixture-recipe.md` to build inside
   `$( … )` with explicit `|| exit 1` guards (not `set -e`, whose
   assignment-from-command-substitution semantics are not dependable across
   shells — this recipe is `bash` but gets pasted into zsh, the macOS default),
   and enter the fixture via `cd "${FIX:?}"`. *Acceptance: the recipe, executed
   verbatim under both bash and zsh with `mktemp` forced to fail from inside a
   real repo, leaves HEAD unchanged; the happy path still builds a fixture.*

## Out of scope

Adopter-repo scripts the kit does not run (check #5b deliberately does not
police them). The broader "empty variable reaches a destructive sink" class
beyond fixture scratch paths: the sweep traced every `rm -rf "$VAR"` (~90),
`git -C "$VAR"` (~40), and `cd "$VAR"` (~60) site and found no other reachable
damage path — `rm -rf ""` is a harmless rc=1 no-op, and no `rm -rf "$X"/*`
shape exists anywhere in the repo.

## Dependencies

None. Mechanism-only; displaces no queued work.

## Verification

Evidence recorded 2026-07-16, all in a sandboxed macOS environment (`/tmp` and
`/var/folders` both denied; only `$TMPDIR` writable) — deliberately the hostile
case, since that is the environment the defect needs:

- **`scripts/test-fixture-isolation.sh` — both directions.** Passes here
  (`[all]` 18 suites, `[nested]` 4 suites, host repo untouched). **Fails on
  `792d46a`** in a detached worktree with four findings:
  `test-install.sh [all]` → leaked commit `init`;
  `hooks/test-session-context.sh [all]` → leaked `first commit`, `second commit`;
  `test-install.sh [nested]` → leaked `init` again, which is what proves the
  `${VAR:?}` layer is load-bearing; and
  `hooks/test-codex-hooks-cwd.sh [all]` → *"reported success (exit 0) though
  every mktemp failed"* — the call-log oracle catching the **vacuous pass** that
  suite had been getting for free, independently of any leak. The commit messages
  are the exact ones that landed on `main` during the session that found this.
  Runtime 8s standalone / 10s nested.
- **check #5b:** zero findings on the fixed corpus (with the bug's own
  description sitting in comments — proving the comment-stripping works); fires
  with a precise message on a reintroduced bare `mktemp -d`; zero findings again
  after restore.
- **`fixture-recipe.md` executed verbatim**, bash **and** zsh, with `mktemp`
  shimmed to fail, from inside the real repo: the guard fires and `HEAD` is
  unchanged in both. Happy path builds a fixture. (The first rewrite of this
  recipe, which relied on `set -e`, **leaked a commit** under zsh during
  verification — that is why the guards are explicit.)
- **The isolation suite's own vacuity guards, sabotage-tested** (each mechanism
  broken independently, against both this tree and the suite as first tagged):
  `chmod -x` on the shim → this tree fails `the mktemp shim is not executable`,
  the first-tagged suite printed a full green (rc=0) having failed zero mktemps;
  `PATH` injection removed → this tree fails `the mktemp shim was never invoked
  across N suite(s)`, the first-tagged suite again green. Control (honest shim)
  passes both modes.
- **Host-repo integrity asserted after every suite run**: `HEAD` unchanged.
- `scripts/test-template-sync.sh` green (33 files); `check-packaging.sh` green
  (v0.18.0); manifest re-pinned to 40 lines with all 6 `# tailored` markers.
- **`bash scripts/verify.sh` — the `harness` gate now passes (`ok: harness`).**
  At v0.17.0 in this same sandbox it FAILED (`FAILED: 8 harness check(s)` — the
  eight bare-`mktemp` hook suites). That gate is the one this release touches,
  and it flipped FAIL → ok. Run directly, the full gate reports
  `OK: agent harness is coherent`.
- **The `evals` and `runtime-fixture` gates still fail here — environment, not
  regression.** Both fail at v0.17.0 in this sandbox too, with the same
  `test-eval: 10 check(s) failed` count as this tree. Cause, confirmed directly
  rather than inferred: with `$TMPDIR` set, `env HARNESS_NESTED_FIXTURE=1 bash
  scripts/check-harness.sh` reports `OK: agent harness is coherent`; with
  `$TMPDIR` **unset**, `${TMPDIR:-/tmp}` falls back to `/tmp`, which this sandbox
  denies, and every suite using that idiom fails — *including* `test-log.sh`,
  `test-verify.sh`, and `test-doc-garden.sh`, which this release never touched
  and which already used the idiom at v0.17.0. The `runtime-fixture` failure is
  the live-runtime port binding, likewise pre-existing.
  **CI is the arbiter for a fully green `verify.sh`; this sandbox cannot produce
  one at any commit.** Note the failure mode is now *loud* (exit 1) where it was
  *silent* (leak): a suite that cannot obtain a safe scratch dir must refuse to
  run — never fall back, and never SKIP into a false green.

## Progress

- 2026-07-16 — Scoped, built, and verified in one session. Found by a
  launch-readiness review; the review's own `verify.sh` runs reproduced it live
  (twice), which is how the mechanism got pinned rather than theorized. A
  five-agent discovery sweep (183 files) then found three things the manual
  read had missed: the `git -C ""` sibling vector; the shipped
  `fixture-recipe.md` teaching the pattern; and the incompleteness of the
  first-pass fix (`|| return 1` inside a command substitution cannot abort its
  caller — `test-install.sh:162`'s `git commit` was still live). Two latent
  v0.17.0 bugs of the same family surfaced while verifying and were fixed here:
  `test-audit-log.sh` and `test-dev-instance.sh` assert "outside a Git worktree"
  behavior in a scratch dir that resolves the **host** repo whenever `$TMPDIR`
  itself sits inside a worktree — now capped with `GIT_CEILING_DIRECTORIES`,
  expressing the invariant *no fixture may discover a Git repository above its
  own scratch base*. `test-codex-hooks-cwd.sh` had the same flaw and was
  **passing vacuously** because of it.
- 2026-07-16 — A four-dimension adversarial review of the candidate diff
  (completeness / gate-correctness / test-validity / release-integrity, each
  finding independently verified by a refuter that had to read or run the code)
  landed three real defects in the first-pass fix, all fixed here:
  - **A surviving instance of the very class being fixed**: an unguarded
    `cd "$work"` at `test-check-harness.sh:1199` running
    `bash scripts/sync-agent-skills.sh` — which writes stubs and `rm -rf`s a
    directory. Same command-substitution gap (`work=$(new_fixture)`) that made
    `test-install.sh:162` live. The sweep had cleared `sync-agent-skills.sh`
    itself as safe, and it is; the caller was the hole.
  - **check #5b shipped with no regression test**, against this repo's own
    "every guard ships with a regression test" convention — the one rule a new
    ERROR-severity gate must not break. Eight cases added to
    `test-check-harness.sh` covering both failure modes, the canonical form, the
    `if !` form, a non-`$TMPDIR` template, non-invocations, the declared
    exception, and the scope boundary.
  - **The isolation test's "mechanism-agnostic" claim was false.** A canary
    holding only `README.md` cannot detect a repo-relative leak
    (`bash scripts/X.sh` dies "No such file or directory" and perturbs nothing) —
    precisely the shape of the line-1199 leak. `new_canary` now copies the
    mechanism in (excluding `test-*.sh`, so a leaked `check-harness.sh` cannot
    re-enter the suite), and the docstring states the real boundary instead of
    overclaiming. The strengthened canary is strictly better: it fails on
    `792d46a` with **four** findings instead of three.
  Three further findings were **refuted** by verification and correctly dropped
  rather than actioned: the "#5b scope under-implements its rationale" claim
  (scope is stated explicitly two sentences above the quoted line), the "heredoc
  limitation documented backwards" claim (the escape hatch matches anywhere on
  the line, not just trailing), and the `local v=$(mktemp …)` dead-guard claim
  (the masking is real and does pass #5b, but no site uses that form and
  shellcheck's SC2155 already covers it).

- 2026-07-16 — **The gate caught its own test file.** Adding the #5b cases made
  #5b fail: five lines in `test-check-harness.sh` were flagged, because the
  fixtures deliberately contain the banned pattern and #5b scans that file too.
  Four were `printf` lines writing bad fixtures; the fifth was an `assert_ok`
  *description string* (`'if ! VAR=\$(mktemp ...)'` — `$(` puts it in command
  position). Resolved by assembling the literal through `MK=mktemp` (an
  assignment, so not command position) rather than reaching for
  `# harness-mktemp-ok`: this file allocates no scratch, and the marker is
  line-scoped and unconditional, so it would also mask a genuinely bad `mktemp`
  added to that line later. That the gate found this on its author is the
  intended behavior working; it is also the honest cost of a text-level check,
  now demonstrated rather than asserted.

## Decisions

- 2026-07-16 — **Explicit `|| exit 1`, never `set -e`, for scratch guards.**
  `set -e` did not abort an assignment-from-command-substitution in one
  reproducible zsh shape (it did in bash, and did in other zsh shapes — the
  boundary was not worth pinning). A guard whose firing condition cannot be
  predicted from reading it is not a guard. The explicit form behaves
  identically in both shells and is already the repo's idiom.
- 2026-07-16 — **Guard at allocation AND at consumption.** Either alone is
  defeatable: allocation guards cannot abort a caller across a command
  substitution, and consumption guards alone leave the failure silent until the
  first `cd`.
- 2026-07-16 — **check #5b is an ERROR, not a WARNING.** It landed on `main`
  twice; a warning would have been ignored the same way the `eval.sh:337`
  comment was — the repo *documented* this exact hazard and still shipped 26
  instances of it. The `# harness-mktemp-ok` escape hatch keeps ERROR severity
  from wedging an adopter.
- 2026-07-16 — **Scope check #5b to what check #6 executes.** What
  check-harness.sh *runs*, it may demand hygiene from; an adopter's other
  scripts are theirs. That boundary is also the blast radius.
- 2026-07-16 — **The oracle must fail when it cannot test.** A 19-agent
  adversarial review found `test-fixture-isolation.sh` could pass **vacuously**:
  `chmod +x "$SHIM/mktemp"` was unguarded, and bash *silently skips* a
  non-executable `PATH` entry and falls through to the real `mktemp` — no error,
  rc=0. Every allocation then succeeds, the canary is never touched, and all
  cases "pass". Reproduced: `chmod -x` on that one file makes the suite as
  tagged print `OK: a failed mktemp aborts every fixture` with rc=0, **having
  failed zero mktemps**. Fixed at three depths: guard the `chmod`, assert the
  `-x` postcondition, and — the general one — require ≥1 real shim interception
  per mode, which catches *any* future cause (a refactored `env` line, an
  absolute-path `mktemp`, shell hashing), not just this one. Kept the per-suite
  `calls == 0` exemption: `test-template-sync.sh` allocates no scratch and has
  nothing to prove. This is the same vacuous-pass class this release caught in
  `test-codex-hooks-cwd.sh`; shipping the *oracle* with its own instance of it
  would have made the release's central evidence unfalsifiable.
- 2026-07-16 — **Declined: `|| return 1` on `work=$(new_fixture)`
  (test-check-harness.sh:1184).** Review proposed it as the other half of the
  :1199 fix. It is one of **76 identical call sites**, none of which check the
  return, and most at top level where `return` is not even valid. This suite
  guards at *consumption* (`${VAR:?}`), which is what stops the leak; the
  load-bearing half shipped. Hardening exactly one of 76 would cargo-cult a
  local fix onto a deliberate design and imply the other 75 were checked.
  Failure mode without it is `bash "/scripts/check-harness.sh"` → 127 → loud.

## Review

A 19-agent adversarial review (4 dimensions: completeness, gate-correctness,
test-validity, release-integrity; each finding independently verified by agents
prompted to refute it) returned **6 confirmed, 9 refuted**. The refuted nine
included three that would have been real regressions if applied — notably a
proposal to tighten the nested-mode selection regex, which would have *narrowed*
coverage in the one direction that fails silently. Folded in: the vacuity fix
above, the recursion-comment correction, and the two documentation overclaims.
Two confirmed findings are deferred, both non-blocking:

- **Perf (~80–110s):** `test-fixture-isolation.sh` is in the install set but not
  in check #6's nested skip list, so it re-runs inside every `test-install.sh`
  fixture (~9 sites × ~10s) and re-proves what the template gate already proved.
  The obvious fix is wrong: `verify.sh` runs the harness gate *with*
  `HARNESS_NESTED_FIXTURE=1`, so adding it to that skip list would also suppress
  the only run of the root/installed copy — the copy that actually leaked. Needs
  a distinct nesting marker.
- **Comment accuracy:** check #5b's honest-boundary note ("does not parse
  heredocs, and `|| true` defeats it") should also name string payloads, now that
  the CHANGELOG and `fixture-recipe.md` document that shape.
