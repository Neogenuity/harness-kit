# Execution sandbox profiles

Status: **active** — v0.16.0 (implementation and evidence complete; maintainer
release actions pending)

## Objective

Ship opt-in, provider-native execution profiles for Claude Code, Codex,
Cursor, and OpenCode, plus an optional authored devcontainer recipe and a
truthful provider-observability map. A repository should be able to adopt the
strongest boundary each provider actually offers, keep that adopted posture
under semantic drift checks, and describe the smallest observed compatibility
weakening for a worktree-owned local runtime without calling it localhost-only
or claiming that permissions, hooks, or a container are equivalent to an OS
sandbox.

## Value

The v0.10.0 baseline documented the safe posture and the v0.15.0 runtime
contract made local applications operable per worktree, but the kit still does
not configure or verify the native controls that are supposed to enforce that
posture. This is now a concrete correctness gap: deleting or weakening a
sandbox config is invisible to `check-harness.sh`, and the queued plan assumed
uniform filesystem/network boundaries that OpenCode does not provide.

The activation review also found that the provider surface moved since the
queue entry was written. Codex now documents native Windows sandboxing and an
experimental domain-aware network proxy; OpenCode added an
`external_directory` approval gate but still has no OS/network sandbox; Claude
can fail closed when its sandbox is unavailable and disable unsandboxed retry;
Cursor distinguishes local sandbox policy from administrator enforcement.
v0.16.0 freezes a freshly verified capability contract before it freezes any
templates or validators.

## Scope

1. **Reverified capability and profile contract.** Recheck every touched
   Execution-containment fact against current primary provider docs and publish
   exact per-provider floors and limits. Define three distinct tiers: a stable
   network-closed baseline; an explicitly weakened compatibility variant for
   the local/private behavior required by a worktree runtime; and separately
   labeled experimental or administrator-enforced options. Keep native Windows
   Codex sandbox support separate from the harness's Bash-hook compatibility
   statement. *Acceptance: every changed fact carries a `verified 2026-07`
   stamp and source; “workspace” is defined honestly as workspace plus any
   isolated temp roots; OpenCode and local Cursor are never described as
   boundaries they do not provide.*
2. **Opt-in provider profiles.** Extend the shipped provider templates with
   the strongest honest repo-local posture: Claude sandbox availability,
   filesystem, network, credential, and unsandboxed-fallback choices; Codex
   workspace-write/on-request with stable network-off defaults and clearly
   experimental scoped loosening; a new `.cursor/sandbox.json`; and OpenCode
   permission rules for shell, web, destructive commands, and external paths.
   Every template includes narrow, reversible loosening guidance and preserves
   existing hand-written configuration. *Acceptance: fresh and existing-config
   fixtures adopt only the requested providers, keep secret deny rules and hook
   wiring intact, and contain no real endpoint, hostname, credential, auth
   header, wildcard public-egress allowance, or raw-prompt telemetry opt-in.*
3. **Declared adoption and semantic drift checks.** Add an optional
   `EXECUTION_PROFILE_PROVIDERS` declaration to `harness.conf`. Unset or empty
   means unadopted and remains valid for legacy installs; once confirmed, the
   declared provider subset is the source of truth and is never inferred from
   surviving config files. `check-harness.sh` validates each declared profile's
   required semantic floor, and `guard-config.sh` covers the new Cursor and
   devcontainer policy paths. Init records explicit choices; update and audit
   propose adoption or report drift without auto-adding or overwriting policy.
   *Acceptance: missing, malformed, disabled, full-access, approval-off,
   broadened-write, and unrestricted-network negative fixtures fail with a
   provider-specific diagnostic; empty/unset and legitimate subsets pass; a
   second update is idempotent.*
4. **Local-runtime compatibility.** Test whether the closed profile and the
   v0.15.0 `dev.sh up|health|seed|down` contract can coexist. A compatibility
   variant may admit the smallest provider-supported bind/connect behavior
   needed for the worktree instance, but must name any broader local/private
   reach and must not silently grant public egress. Audit the v0.15.0 Codex eval
   runner's task-scoped `network_access=true` behavior against the newly
   verified contract and either narrow it or record why it remains explicit
   test-only debt. *Acceptance: exercise two worktrees through startup, health,
   seed, local HTTP, logs, and ownership-safe teardown while probing public
   egress; if the provider cannot express the exact split or the sandbox blocks
   part of the lifecycle, record the observed version/platform limitation and
   ship no stronger template claim.*
5. **Optional devcontainer recipe, authored from the repo.** Follow the
   v0.15.0 contract-over-placeholder decision: init may author a non-root
   `.devcontainer/` bundle only after explicit opt-in and only from a confirmed
   existing image/Dockerfile/Compose topology. It mounts no host credential
   directory or Docker socket, invokes the existing `scripts/dev.sh` lifecycle
   for apps, and is never presented as a provider sandbox or egress policy.
   Existing devcontainers are diff-only. *Acceptance: a representative fixture
   builds and exercises the runtime contract inside the container; otherwise
   this item is formally deferred rather than replaced by an unbuilt universal
   skeleton.*
6. **Provider observability/export map.** Replace the old “point JSONL at
   OTel” assumption with a stamped availability-and-limits table: signal type,
   product/plan scope, configuration location, export mechanism, and privacy
   caveats. Claude and Codex OTel, Cursor hooks/admin analytics/audit, and
   OpenCode local logs/session export are distinct capabilities.
   `.harness/log.jsonl` remains a separate local stream; automatic correlation
   and schema-v2 joins remain in the queued outcome-telemetry plan. *Acceptance:
   every pointer resolves to a current primary source; project-local config
   never carries exporter secrets/endpoints; no provider is credited with an
   OTel or agent-action audit channel its docs do not establish.*
7. **Adoption evidence and doc repair.** Add deterministic schema/semantic
   fixtures plus a behavioral golden task for high-tailoring-cost adoption.
   Repair the shipped pattern's repo-only plan pointer, make installed execution
   guidance self-contained rather than linking to a provider matrix target
   repos do not receive, and remove README's stale claim that no-git
   old-template recovery is still being designed. *Acceptance: the golden task
   preserves existing configs, adopts only selected profiles, keeps the closed
   default and explicitly labels the compatibility weakening, and states
   OpenCode/Cursor limits; all
   changed local links resolve.*

## Out of scope

Reimplementing a provider sandbox or network proxy; claiming a devcontainer is
a sandbox for the host; shipping a telemetry collector, backend, exporter
credentials, or dashboard; organization identity and secrets management;
turning experimental network features into a stable compatibility promise;
native-Windows execution of the kit's Bash hooks; outcome schema v2 and
session-to-PR joins (owned by outcome telemetry); automatically enabling any
profile during update.

## Delegation and integration

The lead owns the interface freeze, this plan/roadmap, root dogfood files,
generated artifacts, version trio, changelog, manifest, integration, and
release. Delegates use disjoint single-writer paths in the shared working tree.

1. **Luna — provider contract and content (single writer).** Own the freshly
   stamped capability/limits table; provider matrix and pattern; init/update/
   audit/router guidance; provider templates; self-contained installed
   execution-profile convention; authored devcontainer recipe; observability
   table; and behavioral eval task. Luna's first deliverable is the exact
   required/optional tuple contract for every profile tier. The lead reviews
   and freezes that contract before Terra writes validators.
2. **Terra — mechanism and deterministic evidence (single writer).** Own
   `harness.conf`, `check-harness.sh`, `guard-config.sh`, any deterministic
   install/update helpers, their regression suites, provider-subset and
   weakening fixtures, and the template-to-root mechanism rollout. Terra
   implements only the frozen tuples and returns any provider ambiguity to the
   lead rather than guessing.
3. **Reciprocal review.** When both branches are green, Terra performs a
   read-only review of Luna's settings for installability, non-clobbering
   adoption, and match to the validator contract. Luna performs a read-only
   review of Terra's mechanism for provider accuracy, false enforcement claims,
   subset behavior, app compatibility, and legacy-update safety. Each author
   fixes confirmed findings only in their owned files.
4. **Lead integration review.** The lead independently verifies every reported
   finding, integrates both streams, adopts the shipped changes into the root
   dogfood harness, runs the inferential `code-reviewer` only after deterministic
   gates pass, and retains sole ownership of release changes and tagging.

## Verification

- **Baseline:** `bash scripts/verify.sh` passed on clean `v0.15.0`
  (`1e96552`) before activation on 2026-07-14.
- **Deterministic profiles:** focused schema/floor, declaration/subset, guard,
  install/update-idempotency, and behavioral-grader fixtures pass. Negative
  cases cover every provider weakening plus Claude excluded-command bypass,
  malformed Codex syntax outside required tuples, duplicate TOML keys/tables,
  and missing Python 3.11+ `tomllib`. The corrected Codex eval runner fixture
  proves all seven task-scoped local/private compatibility overrides, emits a
  visible weakening warning, and leaves default-network tasks closed.
- **Behavioral adoption:** the corrected offline reference passes and all five
  seeded clobber/thin-doc/telemetry shortcuts fail. The first paid Codex
  attempt was invalid measurement evidence: the default workspace-write runner
  could not edit protected `.codex/config.toml`, so it measured runner
  containment rather than adoption behavior and was not baselined. The task now
  explicitly declares `execution: provider-config-write`. After final review,
  real provider runs also require the independent
  `--allow-provider-config-write` CLI flag; metadata or the flag alone refuses
  before invocation. With both present, the runner applies the maintenance
  escape and Codex danger-full posture from a disposable clone but warns that
  the process still has unrestricted host filesystem and public-network access,
  recommends an external container/VM, rejects the incompatible network mode,
  and keeps ordinary tasks unchanged. The corrected Codex run passed 2/3 on
  `gpt-5.6-luna` (`pass@k=1`, `pass^k=0`, rate 0.67; run
  `20260714-v016-provider-config`) and the three-trial cell is baselined. Trial
  2 timed out after missing the substantive convention, so it remains a real
  failure. A single quota-conservative Claude Haiku smoke failed 0/1 after
  changing forbidden OpenCode policy and missing required Claude/convention
  content; it is retained as non-baselined evidence. One stale Claude polling
  subprocess was terminated after its completed gate output could never match
  the sentinel it awaited; no score was reclassified.
- **Cross-review:** Terra and Luna completed reciprocal read-only reviews; all
  confirmed content/mechanism findings were fixed and rerun. The post-gate
  `code-reviewer` found two high issues (Claude excluded commands and partial
  Codex TOML parsing) and one follow-up medium null-type edge; all were fixed
  with regression fixtures, and focused re-review returned no findings. A fresh
  post-live review found two medium documentation inconsistencies in ADR 008 and
  this status section; both were corrected without changing the accepted tuple.
  The final post-evidence review found one high runner-authorization gap:
  repository metadata alone activated Codex danger-full. The independent CLI
  authorization, honest host-impact warning, and refusal tests are the
  resulting correction. A separate pre-commit subagent review then found a
  Claude socket/weaker-isolation false-green and nested Codex proxy-map
  false-greens; the stable tuple, parsed-object checks, dogfood profile, and
  negative fixtures were tightened before publication.
- **Integrated gates:** `bash scripts/verify.sh` passed after the initial
  integration/dogfood rollout, after every initial inferential-review fix, and
  again after the complete live-evidence correction and dogfood re-pin. It also
  passed after the independent danger-full authorization correction; focused
  inferential re-review returned no findings.
- **Live compatibility evidence:** with explicit spend approval, Codex CLI
  0.144.1 (`gpt-5.6-luna`) on macOS rejected concurrent local startup when the
  documented exact localhost map was combined with
  `allow_local_binding = false`. Repeating the same trial with
  `allow_local_binding = true` allowed both worktrees through startup, health,
  deterministic seed, local HTTP, and log assertions; a separate
  `https://example.com/` probe was proxy-blocked with HTTP 403. The latter
  setting admits broader loopback/private-network reach, and the fixture then
  stopped at its first ownership-safe `down` because the workspace-write
  sandbox denied the required `ps` process inspection. A focused follow-up
  confirmed direct signaling was allowed while `ps` was denied, but direct
  signaling is not treated as equivalent ownership proof. The release therefore
  makes no localhost-only or full-lifecycle compatibility claim.
- **Devcontainer:** formally deferred for this non-app repository because recon
  found no confirmed image, Dockerfile, Compose service, or non-root runtime to
  author and build from; no placeholder container was emitted.
- **Release:** `plugins/harness-kit/VERSION`, both plugin manifests, the
  dogfood manifest, and CHANGELOG say `0.16.0`. The complete
  `bash scripts/verify.sh` gate passed on the final release-branch content
  immediately before the release commit; that commit receives the required
  post-commit full run before the annotated `v0.16.0` tag is created.

## Decisions

- 2026-07-14 — **Codex agent TOML follows the provider schema, not generic
  frontmatter:** current custom-agent files accept the required name,
  description, and developer instructions plus supported Codex config keys; the
  canonical provider-neutral tool-name list is not one of those fields. Omit it
  from Codex stubs while retaining it in Markdown-provider stubs.
- 2026-07-14 — **Observed behavior overrides the planned localhost-only tier:**
  the stable Codex profile remains network-off. The accepted opt-in disjunction
  is relabeled an experimental local/private-network compatibility weakening
  and requires `allow_local_binding = true`; it retains the exact
  localhost/127.0.0.1 public-domain map, empty Unix-socket rules, and disabled
  dangerous bypasses. Evidence is stamped to CLI 0.144.1 on macOS, not turned
  into a speculative version gate. Ownership-safe teardown remains an explicit
  workspace-write limitation.
- 2026-07-14 — **A declared Codex profile needs a complete TOML parser:** the
  tuple reader remains the portable semantic checker, but isolated Python 3.11+
  `tomllib` must first accept the whole file. Parser absence is unverifiable and
  fails the declared gate rather than producing false green.
- 2026-07-14 — **Claude excluded commands are not stable-profile local keys:**
  they always run outside the sandbox, including when
  `allowUnsandboxedCommands` is false. Nonempty `excludedCommands` therefore
  blocks adoption instead of being preserved under a stable declaration.
- 2026-07-14 — **This repo formally defers a devcontainer:** no confirmed build
  source exists, and the plan explicitly prefers deferral to an unbuilt
  universal skeleton.
- 2026-07-14 — **Paid evidence remains opt-in:** deterministic config, grader,
  and argv evidence is complete; no live inference trial runs without explicit
  maintainer approval for provider spend.
- 2026-07-14 — **v0.16.0 is the right semver:** provider-template layout and
  shipped check/guard/update behavior change, which is at least minor before
  1.0.
- 2026-07-14 — **Strongest native posture, not fake uniformity:** Claude,
  Codex, Cursor, and OpenCode receive different floors because their actual
  boundaries differ. Missing enforcement is named, never papered over.
- 2026-07-14 — **Adoption is declared once:**
  `EXECUTION_PROFILE_PROVIDERS` distinguishes “declined” from “later deleted”;
  it is confirmed by the user and never inferred from surviving files.
- 2026-07-14 — **Closed, compatibility, and experimental/admin are separate
  tiers:** local runtime needs do not justify public-domain egress, the observed
  broad local/private weakening is named, and an experimental allowlist does
  not become the stable baseline.
- 2026-07-14 — **Contract over universal devcontainer:** init authors from the
  actual repo and an explicit image/topology choice; no all-placeholder
  container template ships merely to make the tree look complete.
- 2026-07-14 — **Provider telemetry and harness JSONL stay separate:** this
  release documents export surfaces and correlation points; the next plan owns
  outcome schema and joins.
- 2026-07-14 — **Delegation keeps single writers:** Luna owns provider truth
  and content; Terra owns mechanism and deterministic assurance; review swaps
  areas; the lead alone integrates and releases.
- 2026-07-11 — **Configure, do not rebuild:** the kit configures native
  enforcement and verifies the adopted floor; it never substitutes a hook for
  a sandbox.
